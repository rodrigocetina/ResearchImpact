① Authentication
Supabase Auth — Login / Session
Faculty log in via email + password or institutional SSO (SAML/OAuth). Supabase returns a signed JWT token stored in the browser. Every subsequent API call carries this token in the Authorization header — the server validates it before touching any data.

Row-level security: faculty can only read/write their own submissions. Admins (Research Office) see all records.

Compliance Considerations

GDPR & Cybersecurity Compliance

This architecture can meet GDPR and cybersecurity standards — but compliance is not automatic. It requires deliberate configuration and institutional sign-off.

✅ Covered Out of the Box
✓ EU data residency — Supabase Frankfurt region (eu-central-1). Personal data never leaves the EU.
✓ Encryption at rest — AES-256 on all Supabase storage. Encryption in transit — TLS 1.2+ enforced by default.
✓ Data Processing Agreement (DPA) — Supabase publishes a GDPR-compliant DPA that UPF-BSM signs as the data controller.
✓ Row-level security — faculty are structurally prevented from accessing each other's data at the database layer, not just in code.
✓ HTTPS everywhere — enforced by Render and Supabase. No plain-text traffic possible.
✓ JWT-based authentication — short-lived signed tokens mean stolen sessions expire quickly.
⚠ Must Be Built Deliberately
! Rate limiting & brute-force protection — API endpoints need request throttling to prevent credential-stuffing attacks on the login flow.
! Security headers — CSP, HSTS, X-Frame-Options, and CSRF protection must be added explicitly to the FastAPI app.
! Secrets management — API keys and DB credentials must live in environment variables, never in source code. GitHub repo must be private.
! Input validation & sanitisation — all form fields validated server-side before DB writes to prevent injection attacks.
! Audit logging — every submission, score change, and admin access should be logged with timestamp and user ID for accountability.
GDPR Obligations — Technology Alone Is Not Enough
📋 Lawful Basis
Processing staff performance data requires a documented lawful basis — typically legitimate interest or employment contract. This must be decided and recorded by UPF-BSM legal, not by the tool.
📄 Privacy Notice
Faculty must be informed at login: what data is collected, why, how long it's kept, who can see it, and how to exercise their rights (access, erasure, portability). A privacy notice must be linked from the tool.
🔍 DPIA Required
A Data Protection Impact Assessment is almost certainly required under GDPR Art. 35 — this tool constitutes systematic monitoring of employees. Must be completed before go-live.
⏱ Data Retention
A policy is needed: how many years do submissions live before deletion? Faculty who leave UPF-BSM — how long is their data kept? Automated deletion or anonymisation must be implemented.
👤 Data Subject Rights
Faculty can request a copy of their data, ask for corrections, or request deletion. A workflow (even a simple email process) must exist to honour these requests within the 30-day GDPR deadline.
🇪🇸 ENS (Spain)
As a Spanish public university, UPF is subject to the Esquema Nacional de Seguridad. Any production system handling staff data must demonstrate at least ENS Basic compliance — requiring documentation, access controls, and incident response procedures.
🟢 Low Risk
The technology itself is sound. Supabase and Render are used by health-tech and fintech companies with stricter requirements than a faculty performance tool. The infrastructure poses no inherent compliance problem.
🟡 Medium Risk
UPF's IT governance policies may require an institutional security review before any new third-party cloud system handling staff data goes live. This process can take weeks and could require changes to the hosting setup.
🔵 Mitigation Path
Engage UPF-BSM IT and the DPO early — before building, not after. If on-premise hosting is required, the same FastAPI + PostgreSQL architecture runs on UPF servers. The code doesn't change, only the hosting layer.
Pre-Launch Compliance Checklist
Sign Supabase DPA — appoints Supabase as a GDPR data processor under UPF-BSM as controller
Select EU region in Supabase project settings (eu-central-1 Frankfurt)
Complete DPIA — required under GDPR Art. 35 for systematic staff monitoring; submit to UPF DPO
Publish faculty privacy notice — linked from the login screen; covers purpose, retention, rights
Document lawful basis for processing (legitimate interest or employment contract) in the Records of Processing Activities (RoPA)
Define data retention policy — set automated deletion or anonymisation schedule in the DB
IT security review — submit architecture to UPF IT for institutional approval before production launch
ENS Basic compliance documentation — access control policy, incident response procedure, backup policy
Implement rate limiting & security headers in FastAPI before any public-facing deployment
Set GitHub repo to private and move all secrets to environment variables before first deployment
